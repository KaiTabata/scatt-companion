1. なぜリバースしたか

SCATT Expert は、射手にとって有用な軌跡データを内部で計算しているにもかかわらず、 その生データを外部に出す API も公式エクスポートも提供していない。 アプリ内で見える指標は限られており、独自の解析(力みの周波数、反動の方向、 心拍と安定度の相関 等)を行うには生サンプル列が必要だった。

~/Library/Application Support/SCATT Electronics/Scatt Expert/storage.dat は SQLite データベース で、これを read-only モードで開けば session/shot のメタデータは 取れる。しかし軌跡(120Hz のサンプル列)は traces.data カラムに 暗号化済み BLOB として格納されており、そのままでは読めない。

本ソフトは、この BLOB を独自リバースで完全復号して使っている。

2. ファイル構造の全体像

storage.dat は標準的な SQLite 3 ファイル。file コマンドで確認できる:

$ file ~/Library/Application\ Support/SCATT\ Electronics/Scatt\ Expert/storage.dat
storage.dat: SQLite 3.x database, last written using SQLite version 3038003

主なテーブル:

  • sessions — 練習セッション (id, position, distance, caliber, sample_rate)
  • traces — 1 trace = 1 shot 相当 (id, session_id, timer, data BLOB)
  • shots — 撃発イベント (trace_id への参照、trace 内オフセット、phase 等)

traces.data 以外は素直に読める。問題は data カラムの中身が 明らかに圧縮 + 暗号化されたバイナリ である点。

3. 最初の観察 (sqlite で見える範囲)

BLOB の最初の数バイトを抜き出して観察すると、こんな見た目:

offset 0: 01 ← 1 byte バージョン (常に 0x01)
offset 1: e3 8a 47 7c f8 c9 ... ← ランダムに見えるバイト列が続く

頭の 0x01 はバージョン番号らしい。続くデータはエントロピーが高く、 ランダムにしか見えない。圧縮 後 暗号化 されているのか、 あるいは 暗号化 後 圧縮 されているのか — 通常、 圧縮後に暗号化するので圧縮済みデータはエントロピーが高い。

ここで重要な仮説: 暗号化が単純な XOR で、その上に qCompress (zlib) が乗っている 可能性が高い。SCATT Expert は Qt で書かれていることが strings コマンドの結果から判明 しており、Qt の qCompress は zlib+4 バイト長前置の独自フォーマット。

4. 復号の 4 段階

結論から書くと、traces.data は次の 4 段階で復号できる:

Aバージョンバイトの確認

先頭 1 バイトが 0x01 であることを確認 → 残りが本体。

BXOR 復号 (CBC 風 chain)

16 バイト固定 XOR 鍵を使って 1 バイトずつ 復号。各バイトは さらに 1 つ前の暗号文バイト とも XOR される (CBC モード風)。

for i, c in enumerate(body):
    out[i] = prev ^ c ^ XOR_KEY[i % 16]
    prev = c   # ← 1 つ前の "暗号文" バイトを次に渡す

これにより、同じデータでも先頭が違えば後ろも変わる (パターンを隠す)。

CqCompress 形式のアンラップ

復号後の最初の 1 バイト + 2 バイト (qChecksum) を捨て、残り 4 バイトを zlib 用の長さヘッダとして スキップし、zlib.decompress() にかける。

raw = zlib.decompress(decoded[1:][2:][4:])

Qt の qCompress フォーマットは 4-byte BE 長さ + zlib stream という独自仕様。 Python 標準の zlib に渡すには長さフィールドを剥がす必要がある。

Dマジックバイト + サンプル列パース

展開後の先頭 4 バイトが 0x0a 0x0b 0x0c 0x0d ならば成功。 その後、offset 12-14 に 2 バイト BE のサンプル数、 offset 15 以降に 12 バイト × N の連続サンプル。

magic = raw[:4]            # b"\x0a\x0b\x0c\x0d"
n = struct.unpack(">H", raw[12:14])[0]
samples = [struct.unpack(">fff", raw[15 + i*12 : 15 + (i+1)*12])
           for i in range(n)]

5. XOR 鍵の特定 — バイナリ静的解析

鍵は SCATT Expert 本体の Mach-O バイナリの __const セクションに 平文で埋め込まれていた。Hopper / Ghidra で逆アセンブルし、storage 文字列の 周辺関数を辿ると XOR ループが見つかり、ループで参照されている定数アドレスを追って 0x1000d6f50 番地 に到達。

そこに 16 バイト連続で並んでいたのが鍵:

XOR_KEY = bytes([
    0xe3, 0x00, 0xe9, 0x00, 0x34, 0x85, 0x1d, 0x04,
    0xf0, 0x95, 0xc0, 0x70, 0x0e, 0x1e, 0xb9, 0xf3,
])

これは本ソフト独自の改変や鍵生成ではなく、SCATT Expert が自身のデータを保存するときに 使っている鍵そのもの。つまりこれを既知のフォーマット仕様として扱う ことに等しい (DRM の迂回ではない)。

6. サンプル形式 — QDataStream の中身

1 サンプル = 12 バイト、全フィールド big-endian:

+0: XX XX XX XX X 座標 (mm) — float32 BE
+4: YY YY YY YY Y 座標 (mm) — float32 BE
+8: CC CC CC CC Cant 角 (radians) — float32 BE

X / Y は SCATT の画面座標系(Y は下方向が正)。Cant は銃身ロール角で、 水平が 0、右に傾けると正。サンプリングレートは sessions.sample_rate から取得 (典型 120 Hz)。

shot との対応は shots.trace_offset カラム — これは trace 内 N 番目のサンプルが 撃発の瞬間に対応する。本ソフトでは samples[:trace_offset] を pre-trigger、 samples[trace_offset:] を post-trigger (反動) として解析する。

7. 完全な Python 復号関数

上記すべてを束ねた実装:

import struct, zlib

XOR_KEY = bytes([
    0xe3, 0x00, 0xe9, 0x00, 0x34, 0x85, 0x1d, 0x04,
    0xf0, 0x95, 0xc0, 0x70, 0x0e, 0x1e, 0xb9, 0xf3,
])

def decode_trace(blob: bytes):
    if not blob or blob[0] != 0x01:
        raise ValueError("bad version byte")
    body = blob[1:]
    # ----- Phase B: XOR 復号 (CBC 風 chain) -----
    out = bytearray(len(body))
    prev = 0
    for i, c in enumerate(body):
        out[i] = prev ^ c ^ XOR_KEY[i % 16]
        prev = c
    # ----- Phase C: qCompress (1B drop + 2B qChecksum + 4B BE length + zlib) -----
    raw = zlib.decompress(bytes(out[1:])[2:][4:])
    # ----- Phase D: マジック + サンプル列 -----
    if raw[:4] != b"\x0a\x0b\x0c\x0d":
        raise ValueError("bad magic")
    n = struct.unpack(">H", raw[12:14])[0]
    return [
        struct.unpack(">fff", raw[15 + i * 12:15 + (i + 1) * 12])
        for i in range(n)
    ]

これがリポジトリ scatt_gui.py の decode_trace() として実装されている関数の全文(コメントを追加した版)。 14 行ほどでフォーマット全体を再現できる。

8. 検証 — 本家との一致確認

正しく復号できているか検証するため、次のような確認をした:

  1. サンプル数の一致: n × 12 + 15 = 展開後の長さに一致するかチェック
  2. shots.trace_offset の位置を取り出して X/Y を表示 → SCATT の表示する着弾点と 画面上で一致 (mm 精度)
  3. 軌跡を本家 SCATT の表示と並べる → 完全に同じ軌跡を再現
  4. Cant 角: 銃を意図的に傾けて撃った trace で、復号した Cant が ±5° 程度の差で本家の表示と一致

これにより、復号は 意図したサンプル列を 100% 再現していると判断できた。
