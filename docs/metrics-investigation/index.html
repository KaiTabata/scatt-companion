1. 調査の目的と方法

本ソフト (SCATT Companion) の指標値を、本家 SCATT Expert の表示と 完全に同じ意味にするための調査記録。「同じ shot で同じ数字が出る」状態を目指した。

方法:

  1. SCATT_MX-02 公式 PDF マニュアル を読み、各指標の定義文を抽出
  2. SCATT Expert の Mach-O バイナリを otool + llvm-objdump で静的解析、UI 文字列のクロスリファレンスから関連関数を推定
  3. 同一の storage.dat を本ソフトと SCATT 本家の両方で開き、特定 shot の数値を比較

2. 情報源

  • SCATT_MX-02 USER MANUAL (公式 PDF、英語、29 ページ) — General Info, Distance curves, Coordination curve, Aiming point speed curve, Options menu, Shots parameters menu
  • SCATT Expert binary (/Applications/Scatt Expert.app/Contents/MacOS/Scatt Expert, Mach-O x86_64, 7.4MB) — strings, symbols, x86 disassembly
  • 実データ storage.dat Session #6 (50m prone, 90 shots, 2026-05-12)
  • 本家 SCATT 画面のスクリーンショット — Series 1 の shot 1-9 の数値を視認

3. 確定済みの指標

S1 — 直前 1 秒の平均照準速度 (mm/s)

計算式が確定。本家との誤差 ≤ 1 mm/s で完全一致。

定義 (PDF + 実測):

S1 = meank ∈ 直前 1 秒 |v[k]| [mm/s]

v[k] = √((x[k+1]−x[k])² + (y[k+1]−y[k])²) × fs

fs = 120 Hz (典型値、session ごとに sessions.sample_rate から取得)

実装: scatt_analysis.mean_velocity_last(t, 1.0)

10a / 10a-0.5 / 10b / 9c (滞在率 %)

計算式が確定。本家と一致。

定義 (PDF page 18 "Average steadiness in 10.0"):

"the amount of the final analysis (control) time up to the moment of shot release that the aiming point was within the 10.0, expressed in percentages"

計算式:

10a = Σk ∈ 直前 1 秒 1[ r[k] ≤ R10 ] / Nwindow × 100 [%]

r[k] = √(x[k]² + y[k]²) (中心からの距離)
R10 = 5.2 mm (50m ライフル) / 0.25 mm (10m AR)

DA — 平均標準点 (MPI) からの距離 (mm)

定義が確定。実装は session 集計時に注入。

定義 (PDF page 18 "Accuracy of shooting"):

"the average point from the points described above is calculated and its distance from the center of the target is measured"

計算式:

MPI = (mean(xfire), mean(yfire)) (session 全 shot の平均着弾点)

DA[shot] = √((xfire[shot] − MPIx)² + (yfire[shot] − MPIy)²) [mm]

4. 部分的に判明 (未完全)

S2 — 直前 250ms の照準速度 (mm/s)

単位と概念は判明。完全一致する計算式は特定できず、近似実装中。

判明していること:

  • 単位は mm/s (ユーザ確認済)
  • 窓は 250ms (ユーザ確認済)
  • 意味は「短期の照準速度」

判明していないこと:

  • 窓の 正確な終端 — trace_offset と一致する shot もあれば、最大 480ms ほど前で測ったほうが合う shot もある
  • 窓内で全サンプルを平均するのか、何らかの 静止区間検出を併用するのか不明

検証データ (Session #6):

247.229.347.2p25 が完全一致

11 種類の式 (単純平均 / 窓ずらし / rolling min / rolling p25 / 重み付き / EWMA / displacement) を試した結果、rolling p25 (直前 1 秒内で 250ms rolling mean の 25 パーセンタイル) が最小 RMSE 5.16 で最も近い。これを近似値として採用。

実装: scatt_analysis.s2_rolling_p25(t)

5. 未解明

SCATT バイナリの該当関数を完全に逆解析するには Hopper Pro や Ghidra が必要。

難しい理由:

  • SCATT Expert は ストリップ済み Mach-O バイナリ — 内部クラス Result や Analysis のシンボルが消されている
  • 公開シンボルは Qt の MetaObject 関連 (CalibrationAiming, HardwareData, ScattMXBase) のみ
  • "intervalStability" などの Qt property name は埋め込み文字列として残るが、その property の setter が誰かを追うのに数十のクロスリファレンス追跡が必要
  • SCATT は速度計算を inline している可能性が高く、関数境界自体がはっきりしない

取れる手段:

  1. Hopper Disassembler ($129) の Pseudo-code 機能で C 風コードを表示し追跡
  2. Ghidra (NSA 開発、無料) で関数境界を自動推定 + decompile
  3. lldb で実行中の SCATT にアタッチし、property setter にブレークポイントを置いて呼び出し元を辿る
  4. SCATT Electronics に問い合わせ — S2 の式を info@scatt.com に直接質問

6. 検証手法の詳細

バイナリ静的解析の経過

SCATT 本家バイナリ内に見つかった関連文字列:

アドレス文字列用途
0x100679bb4"Stability of time interval between shots"UI 表示
0x100679bf2"Stability of aiming"UI 表示
0x1006a8e8e"intervalStabilityChanged"Qt signal name
0x1006a8ea7"intervalStability"Qt property name

"Stability of aiming" は 0x100271e7e の lea rdx, [rip + 0x407d6d] 命令から参照されている。周辺コードは Qt Property System への登録 (UI 表示用バインディング) を行っており、計算自体は inline された別関数で行われている模様。

実データ照合の経過

Session #6 (sid=6, 50m prone, 90 shots, 2026-05-12) の最初 9 shot について、本家 SCATT の表示と本ソフトの計算を直接比較。

  • S1: 9/9 shot で誤差 ≤ 1.4 mm/s (完全一致と判断)
  • S2: 試行錯誤の結果、rolling p25 が最良 (RMSE 5.16) だが特定の shot で 5-10 mm/s ずれが残る

7. 推奨アクション

本ソフトの実装方針:

  • S1 は本家完全互換として使用
  • S2 は近似値として表示し、ツールチップで「本家と多少差あり」と明示
  • R95 (mm, 円の半径) は別指標として残す — 円の広がりを見る用途で有用
  • DA は session 集計時に MPI を計算して各 shot に注入

正確化のための今後の手段:

  1. SCATT バイナリの Hopper / Ghidra による decompilation
  2. SCATT サポートへの問い合わせ
  3. SCATT 公式 Wiki / フォーラム (ロシア語) で正確な式の調査

注: SCATT 本家のアルゴリズムが Black Box でも、本ソフト独自の指標 (発射前後の R95 比、フォロースルー、力みエネルギー、心拍連携) は本家にない分析を提供できる。S2 完全一致は「数字を揃える」ためだけの作業であり、機能的には他の指標で代替可能。
